# Contributing

This is a teaching example, but the contribution flow mirrors how a real Team OS would work. If you spot bugs or want to suggest improvements:

## How to contribute

1. Fork the repo
2. Create a branch (`git checkout -b improve-{thing}`)
3. Make your edit
4. Open a PR with:
   - What you changed
   - Why (1-2 sentences)
   - Anything you're unsure about

## Quality bar (lightweight)

- No API keys or secrets in files
- README/SKILL.md updated if behavior changes
- Owner is clearly named (frontmatter or top of file)
- One canonical version per concept (don't add a 3rd customer summary template — improve the existing one)

## Coaching, not gatekeeping

If you submit a PR with an issue (e.g., a hardcoded API key, a missing owner), you'll get a comment with how to fix it — not a rejection. Help yourself help others. Friction kills shared resources.

## Things to flag

- Skills with no owner
- Rules that contradict other rules
- Files that look stale (last reviewed > 6 months ago)
